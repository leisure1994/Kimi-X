# Kimi-Agent 未来路线图

> 本文档列出 Kimi-Agent 各版本的功能状态，包括**已完全实现**、**已设计但未完全实现**、以及**未来设想**的功能。
>
> 优先级定义：
> - **P0 (Critical)** — 核心功能，当前版本已实现
> - **P1 (High)** — 重要功能，已设计或部分实现
> - **P2 (Medium)** — 有价值的功能，已设计但未实现
> - **P3+ (Future)** — 未来设想，待后续版本规划

---

## 已完全实现（P0）— 当前版本 v1.0.0-beta

以下功能已在当前版本中完整实现并通过测试（230 个测试全部通过）。

### 项目基础

- [x] **项目骨架和配置系统** — 完整的包结构、pyproject.toml、YAML 配置管理、环境变量支持
- [x] **CLI 入口** — Typer 驱动的命令行界面，支持所有工作模式和配置命令
- [x] **TUI 界面** — 基于 Rich 的终端界面，支持流式输出、语法高亮、实时成本显示
- [x] **认证管理** — API Key 配置（环境变量/命令行/配置文件），支持多配置文件

### LLM 客户端

- [x] **LLM 客户端** — Kimi API 完整客户端（OpenAI 兼容），支持异步流式调用
- [x] **流式处理** — SSE 事件解析、流式事件聚合、thinking/content/tool_call 分离
- [x] **成本追踪** — 实时 Token 计数（tiktoken）、成本计算、预算管理、缓存命中率分析
- [x] **错误处理** — 完整的错误类型体系（认证/限流/超时/网络/Token超限），指数退避重试机制

### 工具系统

- [x] **工具注册表** — 20+ 工具的自动发现和注册，OpenAI Schema 格式转换
- [x] **文件操作工具** — 8 个文件工具（读取/写入/编辑/补丁/目录列表/正则搜索/文件名搜索/文件信息），含沙箱路径检查
- [x] **Shell 工具** — 安全 Shell 执行，内置危险命令黑名单（15+ 危险模式）、敏感命令检测、超时控制
- [x] **Git 工具** — 4 个 Git 工具（状态/差异/日志/分支），基于 GitPython
- [x] **Web 工具** — 网页搜索（DuckDuckGo）和 URL 内容获取，支持 HTML 清理和文本提取
- [x] **子 Agent 工具** — 3 个子 Agent 管理工具（创建/查询/关闭），支持 8 种角色
- [x] **系统工具** — 桌面通知（跨平台）和任务列表管理

### 核心引擎

- [x] **Agent 引擎** — 完整的主循环实现，支持认知分析、模式决策、记忆检索、LLM 调用、工具执行
- [x] **事件系统** — 8 种事件类型（thinking/content/tool_call/tool_result/mode_switch/cost_update/done/error）
- [x] **回合管理** — 单次交互的生命周期管理（创建/执行/完成/错误）
- [x] **上下文管理** — 消息构建、历史维护、系统提示词动态调整

### 工作模式

- [x] **5 种工作模式** — Explore（探索）、Plan（规划）、Agent（交互执行）、Auto（自适应）、YOLO（全自主）
- [x] **认知-决策分离架构** — ModeRouter 实现任务特征分析、权重矩阵决策、风险限制
- [x] **模式切换** — 运行时模式切换，AUTO 模式下的自动路由决策

### 记忆系统

- [x] **记忆管理器** — 统一的多维记忆接口（recall/store/consolidate）
- [x] **工作记忆** — 短期缓存，支持文件内容 LRU 淘汰、工具结果缓存、变量空间
- [x] **情景记忆** — SQLite 持久化，FTS5 全文搜索，事件记录和检索
- [x] **语义记忆（基础）** — 数据模型和接口定义，数据库表已创建

### 子 Agent 系统

- [x] **子 Agent 编排器** — 32 并发调度（可配置到64），asyncio.Semaphore 控制
- [x] **工作进程** — 独立的子 Agent 执行引擎
- [x] **结果收集** — asyncio.Queue 结果队列，支持 wait_all/wait_any/cancel
- [x] **8 种角色定义** — explorer/planner/coder/reviewer/tester/debugger/researcher/documenter

### 测试与质量

- [x] **完整测试套件** — 230 个测试全部通过
  - 12 个单元测试模块（引擎/记忆/工具/配置/成本/LLM/子Agent/模式/沙箱）
  - 1 个集成测试模块（Agent 完整工作流）
  - pytest-asyncio 异步测试支持
  - pytest-cov 覆盖率报告

## 已设计但未完全实现（P1-P2）

以下功能已有完整的架构设计和接口定义，部分代码已实现，但需要进一步开发才能投入生产使用。

### 语义记忆知识图谱（P1）

**状态**: 架构设计完成，数据库表已创建，自动构建逻辑待完善

**已实现的**: 
- `SemanticMemory` 类框架和接口定义
- SQLite 数据库表结构（knowledge_entries、project_maps 等）
- `build_project_map()` 方法框架

**待实现的**:
- [ ] 代码库自动扫描和实体提取（文件、类、函数、模块）
- [ ] 关系自动发现（import 依赖、调用关系、继承关系）
- [ ] 知识图谱增量更新机制
- [ ] 向量嵌入生成和相似度计算

**技术方案**: 基于 AST（抽象语法树）解析 Python 代码，提取类和函数定义，构建实体关系图。使用 sqlite-vec 或 numpy 实现本地向量搜索。

### MCP 客户端（P1）

**状态**: 架构预留，具体协议实现待完成

**已实现的**:
- 配置文件中的 MCP 服务器配置结构
- CLI 命令接口（`kimix mcp list/add/validate`）

**待实现的**:
- [ ] MCP（Model Context Protocol）协议客户端实现
- [ ] MCP 服务器连接管理（stdio/SSE）
- [ ] 工具发现和能力协商
- [ ] 资源访问和 Prompt 模板支持

**技术方案**: 参考 Anthropic MCP 规范，实现 stdio 和 SSE 两种传输方式的客户端，支持工具动态发现和调用。

### 多模态感知（P2）

**状态**: 图像/文档理解接口预留，集成待完成

**已实现的**:
- LLM 客户端中多模态消息的数据模型
- 配置文件中的多模态设置项

**待实现的**:
- [ ] 图像输入处理（截图分析、UI 审查、错误截图解读）
- [ ] 文档解析（PDF/Word/Excel 直接读取和理解）
- [ ] 视频分析（录屏内容分析）
- [ ] 图像生成调用

**技术方案**: 利用 Kimi K2.6 的原生多模态能力，通过 OpenAI 兼容的 image_url 接口传递图像数据，文档解析使用 pdfplumber + python-docx。

### LSP 诊断集成（P2）

**状态**: 设计完成，未实现

**已实现的**:
- 无

**待实现的**:
- [ ] LSP 客户端实现（基于 pylspclient）
- [ ] 代码诊断信息获取（错误、警告、提示）
- [ ] 代码补全建议集成
- [ ] 定义跳转和引用查找

**技术方案**: 集成 python-lsp-server 作为 LSP 后端，通过 JSON-RPC 协议获取代码诊断信息，在 Agent 决策时提供代码质量参考。

### 会话 fork/restore 完整实现（P2）

**状态**: 基础框架存在（Session 和 SessionManager），完整 fork/restore 待实现

**已实现的**:
- `Session` 数据模型和基础 CRUD
- `SessionManager` 的 create/save/load/list 方法

**待实现的**:
- [ ] 会话 fork（从任意回合分叉新会话）
- [ ] 会话 restore（恢复到指定历史状态）
- [ ] 会话分支可视化
- [ ] 会话快照和压缩存储

**技术方案**: 在现有 SQLite 存储基础上，增加会话分支树结构，支持 copy-on-write 的会话分叉。

## 未来设计设想（P3+）

以下功能是团队的未来设想和方向性规划，尚未进入详细设计阶段，将在后续版本中逐步评估和实现。

### 智能知识图谱 Auto-Build（P3）

**描述**: 自动分析代码库构建实体关系图，支持增量更新。

**设想功能**:
- [ ] 自动扫描代码库，提取类、函数、变量、模块等实体
- [ ] 分析实体间的关系（调用、继承、依赖、包含）
- [ ] 生成可视化的知识图谱（支持 Graphviz/Mermaid 输出）
- [ ] 代码变更时自动增量更新图谱
- [ ] 基于图谱的智能代码导航和推荐

**价值**: 让 Agent 深度理解项目结构，提供更精准的代码分析和建议。

### 自适应学习系统（P3）

**描述**: Agent 从交互中学习用户偏好，自动调整行为。

**设想功能**:
- [ ] 用户编码风格学习（命名习惯、代码格式、设计模式偏好）
- [ ] 常用工具和工作流识别
- [ ] 自动调整审批阈值（根据用户历史操作习惯）
- [ ] 个性化提示词模板生成
- [ ] 错误模式学习（避免重复犯同样错误）

**价值**: Agent 越用越懂用户，提供更个性化的开发体验。

### 团队协作模式（P4）

**描述**: 多用户共享记忆和上下文，支持团队协作。

**设想功能**:
- [ ] 共享项目记忆（团队级知识图谱）
- [ ] 会话共享和协作编辑
- [ ] 团队编码规范统一执行
- [ ] 代码审查工作流集成
- [ ] 权限管理和操作审计

**价值**: 从个人 Agent 升级为团队智能助手。

### Plugin 市场（P4）

**描述**: 社区贡献工具和技能包，可插拔扩展。

**设想功能**:
- [ ] 工具包发布和分发机制
- [ ] 技能包（Skill Pack）定义规范
- [ ] 社区评分和版本管理
- [ ] 一键安装和更新
- [ ] 官方认证和审核流程

**价值**: 开放的生态体系，让社区共同丰富 Agent 的能力。

### Web UI（P4）

**描述**: 浏览器版管理界面，提供更丰富的交互体验。

**设想功能**:
- [ ] 会话历史浏览器（支持搜索和过滤）
- [ ] 代码编辑和预览（集成 Monaco Editor）
- [ ] 知识图谱可视化
- [ ] 成本分析仪表盘
- [ ] 设置和配置界面

**价值**: 超越终端的限制，提供更现代的交互体验。

### CI/CD 集成（P3）

**描述**: GitHub Actions / GitLab CI 原生支持。

**设想功能**:
- [ ] GitHub Actions 官方 Action
- [ ] GitLab CI 组件
- [ ] 自动化代码审查（PR/MR 评论）
- [ ] 自动化测试生成和执行
- [ ] 部署流水线集成

**价值**: 将 Agent 能力嵌入开发流水线，实现自动化代码质量保障。

### 语音交互（P5）

**描述**: 语音输入输出支持。

**设想功能**:
- [ ] 语音输入（STT，Speech-to-Text）
- [ ] 语音输出（TTS，Text-to-Speech）
- [ ] 语音命令识别
- [ ] 实时语音对话模式

**价值**: 解放双手，提供更自然的交互方式。

### 实时代码补全（P4）

**描述**: IDE 级别的实时代码建议。

**设想功能**:
- [ ] 基于上下文的代码补全
- [ ] 函数签名自动推断
- [ ] 文档字符串自动生成
- [ ] 代码重构建议实时提示

**价值**: 将 Agent 能力延伸到编码的每一个瞬间。

### 智能代码索引（P3）

**描述**: 基于 AST 的代码结构索引，加速代码理解和搜索。

**设想功能**:
- [ ] AST 解析和索引构建
- [ ] 符号搜索（类/函数/变量定义和引用）
- [ ] 代码结构可视化
- [ ] 变更影响分析

**价值**: 让 Agent 在毫秒级别理解代码库结构。

### A/B 测试框架（P4）

**描述**: 不同提示策略的效果对比。

**设想功能**:
- [ ] 多提示词模板对比执行
- [ ] 效果评估指标（准确率/成本/速度）
- [ ] 自动化测试集执行
- [ ] 结果报告和推荐

**价值**: 数据驱动地优化 Agent 行为。

### 自动化文档生成（P3）

**描述**: 从代码自动生成技术文档。

**设想功能**:
- [ ] 代码注释自动生成和补全
- [ ] API 文档自动提取（支持 OpenAPI/Swagger）
- [ ] README 和架构文档自动生成
- [ ] 变更日志自动编写

**价值**: 减少文档维护负担，保持文档与代码同步。

### 安全审计模式（P3）

**描述**: 自动检测代码中的安全漏洞。

**设想功能**:
- [ ] 常见安全漏洞模式检测（SQL注入/XSS/CSRF/缓冲区溢出）
- [ ] 依赖漏洞扫描（集成 safety/pip-audit）
- [ ] 密钥和敏感信息泄露检测
- [ ] 安全修复建议自动生成

**价值**: 在开发阶段发现并修复安全问题。

## 如何贡献

如果你对这些未来功能感兴趣，欢迎通过以下方式参与：

1. **提交 Issue** — 描述你需要的功能或遇到的问题
2. **提交 PR** — 实现上述 P1-P2 功能（请先开 Issue 讨论设计）
3. **分享反馈** — 告诉我们哪些 P3+ 功能对你最有价值，帮助我们排定优先级

> 注意：P3+ 功能目前仅为方向性设想，具体实现方案可能会在后续版本中调整。如果你希望某个功能提前实现，请在 Issue 中说明你的使用场景，我们会根据社区反馈调整优先级。
