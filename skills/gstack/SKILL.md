# gstack — AI 软件公司角色分工

## 📌 技能概述

gstack 解决 "谁来干活" 的问题。

**Superpowers 解决 "怎么干活"。**
**gstack 解决 "谁来干活"。**

**角色分工：**
- **CEO**: 需求审查 — 商业价值、用户痛点、优先级
- **设计师**: 体验审查 — UI/UX、交互流程、一致性
- **工程经理**: 架构审查 — 技术选型、扩展性、依赖风险
- **QA**: 质量审查 — 测试覆盖、边界情况、异常处理
- **安全负责人**: 安全审查 — 注入、越权、数据泄露、加密
- **发布负责人**: 交付审查 — 部署流程、回滚方案、监控

---

## 🎭 角色定义

### CEO（需求审查员）
**关注：**
- 这个需求解决什么用户痛点？
- 商业价值是什么？
- 有没有更简单的方案？
- 优先级合理吗？

**输出：** 需求评估报告（通过 / 需澄清 / 驳回）

### 设计师（体验审查员）
**关注：**
- 交互流程是否自然？
- 错误提示是否友好？
- 是否符合设计系统？
- 可访问性（Accessibility）

**输出：** 体验评估报告

### 工程经理（架构审查员）
**关注：**
- 技术选型是否合适？
- 扩展性如何？
- 依赖是否过多？
- 性能瓶颈在哪？

**输出：** 架构评估报告

### QA（质量审查员）
**关注：**
- 测试覆盖是否充分？
- 边界情况处理了吗？
- 异常路径测试了吗？
- 兼容性如何？

**输出：** 质量评估报告

### 安全负责人（安全审查员）
**关注：**
- 输入验证了吗？
- 有注入风险吗？
- 权限控制正确吗？
- 敏感数据加密了吗？

**输出：** 安全评估报告

### 发布负责人（交付审查员）
**关注：**
- 部署流程自动化了吗？
- 有回滚方案吗？
- 监控告警配置了吗？
- 文档更新了吗？

**输出：** 交付评估报告

## 🔄 工作流程

```
需求提交
    ↓
CEO 审查 → [通过 / 需澄清 / 驳回]
    ↓
设计师审查 → [通过 / 修改建议]
    ↓
工程经理审查 → [通过 / 架构调整]
    ↓
开发实现
    ↓
QA 审查 → [通过 / Bug 清单]
    ↓
安全审查 → [通过 / 安全修复]
    ↓
发布审查 → [通过 / 交付准备]
    ↓
上线
```

**并行审查：** 设计师、工程经理、QA 可以并行工作。

## 💡 使用示例

```
用户: "帮我做一个用户登录功能"

Agent (gstack 激活):
  CEO: "确认商业价值 — 这是核心功能，优先级 P0，通过"
  设计师: "体验建议 — 错误提示用中文，支持密码显示切换"
  工程经理: "架构建议 — 用 JWT + bcrypt，支持刷新 token"
  
  [开发实现...]
  
  QA: "测试覆盖 — 成功登录、失败登录、token 过期、SQL 注入"
  安全: "安全建议 — 密码最小 8 位，限流防暴力破解"
  发布: "交付检查 — 需要 API 文档、部署配置、监控"
```

## 🔧 集成到 Kimi-Agent

```python
from kimix.skills import SkillRegistry

# 注册 gstack
registry = SkillRegistry()
registry.load_skill("gstack")

# 在代码审查时自动多角色审查
if task_type == "code_review":
    roles = ["qa", "security", "release"]
    for role in roles:
        review = registry.get_skill("gstack").execute_role(role, code)
        print(f"[{role}] {review}")
```
